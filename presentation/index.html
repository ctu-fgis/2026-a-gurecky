About

What is GRASS and why contribute?

GRASS (Geographic Resources Analysis Support System) is one of the oldest and most powerful open-source GIS platforms in the world, developed since 1982 and used by researchers, governments, and practitioners worldwide.

As part of the Free software GIS course at FCE CTU, we were challenged not just to use open-source tools, but to actually become part of them. That means reading codebases written by strangers, studying contributor guides, and patiently waiting for maintainers on the other side of the world to review your work.

It is a long track, but every merged PR is a small piece of software that will outlive the semester.

grass-addons PR #1682

r.lake.series: Parallelization of the tool

r.lake.series simulates rising water levels over a series of elevation thresholds, which is computationally expensive on large rasters. This PR added multiprocessing support so the iterations can run in parallel, significantly reducing runtime on multi-core machines. A deeper dive into GRASS's Python scripting API and the challenges of parallelising raster workflows safely.

Takeaways

What open source actually teaches you

🗺️

Navigating a large codebase

You learn to read code you didn't write, trace execution paths across dozens of files, and build a mental model without anyone holding your hand.

💬

Code review is dialogue

Maintainers don't just approve. They push back, ask "why", and sometimes redirect you entirely. You learn to accept that your first instinct isn't always right.

⏳

Maintainers are also human

Reviews take weeks, not hours. Maintainers are volunteers with jobs and lives outside this project. Following up politely without being pushy is a real art.

📐

Style and conventions matter

Every project has a contributor guide, and then some rules that only emerge from reading the code. Naming conventions, test expectations, commit message formats. Matching them signals that you respect the community's work.

🔬

Test, test and test

Production code is used in ways you can't predict. Writing changes that are backward-compatible and don't break existing workflows requires careful thinking and proper testing.

🌍

You're part of something bigger

GRASS has been developed since 1982. Every file you touch has been shaped by dozens of contributors over decade. Your fix runs on machines in research labs, government agencies, and universities across the world. That feeling doesn't get old.

Before this course, I used open-source software as a black box. After five pull requests, I understand it as a living community: imperfect, sometimes slow, but collaborative and reliable for being made in the open.

If you're considering contributing to open source: start small, read before you write, be patient with the review process, and don't get discouraged when maintainers ask for changes. The moment your PR gets merged is worth every revision.
